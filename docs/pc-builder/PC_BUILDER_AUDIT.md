## 1. Визуальные проблемы

### BUG-1: `totalPrice` не мемоизирован корректно в зависимостях
**Тип:** Визуальная/логическая ошибка
**Место:** `src/frontend/src/hooks/usePCBuilder.ts:562-596`
**Проблема:** `totalPrice` зависит от `selectedComponents` целиком, но `useMemo` перечисляет каждое поле отдельно. Если объект `selectedComponents` меняет ссылку (а он меняется при каждом `selectComponent`/`removeComponent`), мемоизация сбрасывается. Это не баг в строгом смысле, но приводит к **ненужным пересчётам** при изменении несвязанных полей.
**Рекомендация:** Зависеть от `selectedComponents` целиком или использовать отдельные memo для каждого ценового узла.

### BUG-2: Периферийные слоты не имеют `onClear` для `incompatible` состояния
**Тип:** Визуальная/функциональная
**Место:** `src/frontend/src/pages/PCBuilderPage/PCBuilderPage.tsx:572`
**Проблема:** Для периферии (monitor, keyboard, mouse, headphones) `onClear` вызывается только при `state === 'selected'`, но НЕ при `state === 'incompatible'`. В отличие от основных слотов (строка 443), периферийные слоты могут оказаться в состоянии `incompatible` через глобальную проверку совместимости, но у пользователя нет видимого пути сбросить такой компонент.
**Рекомендация:** Добавить `|| slotState.state === 'incompatible'` в условие `onClear`.

### BUG-3: Priority-слоты имеют `transform: scale(1.02)` что конфликтует с `transform: translateY(-1px)`
**Тип:** Визуальная
**Место:** `src/frontend/src/components/pc-builder/ComponentSlot.css:23-27`
**Проблема:**
```css
.component-slot--priority { padding: 24px; transform: scale(1.02); }
.component-slot--priority:hover { transform: scale(1.02) translateY(-1px); }
```
При этом `framer-motion` также применяет `transform` через `animate` для `motion.div`. **Трансформации конфликта**: CSS `transform` переопределяет анимацию framer-motion и наоборот. Видимый результат может «прыгать» при взаимодействии.
**Рекомендация:** Перенести scale на framer-motion `animate`, убрать `transform: scale(1.02)` из CSS.

### BUG-4: Tooltip описания обрезается в мобильной зоне
**Тип:** Визуальная/UX
**Место:** `ComponentSlot.css:350-374`
**Проблема:** `component-slot__tooltip` позиционирован `position: absolute; bottom: 100%; left: 50%; transform: translateX(-50%)`. При `max-width: 280px` tooltip может выходить за пределы экрана на мобильных. `max-width` не адаптируется под viewport.
**Рекомендация:** Добавить медиа-запрос `@media (max-width: 768px) { .component-slot__tooltip { max-width: calc(100vw - 40px); } }`.

## 2. Проблемы React компонентов и анимаций

### BUG-5: `motion.div` использует `key={name}` — полная перерисовка при изменении имени
**Тип:** Анимация/производительность
**Место:** `ComponentSlot.tsx:161-168`
**Проблема:** `key={name}` заставляет React полностью уничтожить и пересоздать DOM-элемент при каждом изменении названия компонента. `AnimatePresence` не используется для этого элемента (только для `bsp__component-list` в BuildSummaryPanel), поэтому `exit` анимация не срабатывает — элемент просто «прыгает».
**Рекомендация:** Заменить `key` на стабильный ID или убрать `key`, заменив на `initial` + `animate` как в остальных случаях.

### BUG-6: `motion.div` с `key={price}` аналогично перерисовывает цену
**Тип:** Анимация/производительность
**Место:** `ComponentSlot.tsx:190-201`
**Проблема:** Та же проблема — `key={price}` вызывает полный переконструкт при каждом изменении цены.
**Рекомендация:** Использовать `animate` для интерполяции числа без пересоздания DOM.

### BUG-7: `layout` prop на `motion.div` без `layoutId` вызывает лишние расчёты
**Тип:** Производительность
**Место:** `ComponentSlot.tsx:113`, `BuildSummaryPanel.tsx:191`
**Проблема:** `layout` заставляет framer-motion вычислять изменение размеров/позиции при каждом ререндере. Для списка с ~16 слотами это создаёт ~16 layout calculations на каждый ререндер.
**Рекомендация:** Использовать `layout="position"` или убрать `layout` если не требуется анимация изменения позиции.

### BUG-8: `AnimatePresence` без `key` на родительском элементе
**Тип:** Анимация
**Место:** `BuildSummaryPanel.tsx:183`
**Проблема:** `AnimatePresence mode="popLayout"` применяется к списку `<ul>`, но сам `<ul>` не имеет `key`, поэтому AnimatePresence не отслеживает вход/выход списка целиком — только дочерние `motion.li`.
**Рекомендация:** Вынести `AnimatePresence` оборачивая каждый `motion.li` индивидуально, или добавить `key` на `<ul>`.

## 3. Проблемы совместимости

### BUG-9: Storage (накопители) — НЕТ проверки совместимости вообще
**Тип:** Проверяемая ошибка, блокирующая
**Место:** `CompatibilityService.cs`, `usePCBuilder.ts`
**Проблема:** Нет проверки M.2 слотов на материнской плате vs количество NVMe дисков. Нет проверки SATA портов vs SATA диски. Хранилище полностью игнорируется в обеих проверках (клиентской и серверной).
**Рекомендация:** Добавить проверку `m2Slots` и `sataPorts` в спецификации материнской платы и соответствующую проверку в `CompatibilityService` и `compatibilityUtils`.

### BUG-10: Дубликат группы `tr4` в frontend `compatibilityRules.json`
**Тип:** Данные
**Место:** `src/frontend/src/config/compatibilityRules.json:222-236`
**Проблема:** Группа `tr4` определена дважды: сначала `["TR4"]`, затем `["TR4", "TRX4"]`. Вторая копия добавляет TRX4 в группу без документального обоснования. Backend JSON не имеет этой дубликации.
**Рекомендация:** Синхронизировать frontend JSON с backend JSON или документировать разницу.

### BUG-11: Проверка RAM-слотов на материнской плате не реализована на frontend
**Тип:** Несоответствие клиент/сервер
**Место:** `usePCBuilder.ts:168-249`
**Проблема:** Backend `CompatibilityService:272-277` проверяет `ram.Modules > mb.RamSlots` (Error). Frontend `checkBuildCompatibility` в `usePCBuilder.ts` НЕ проверяет количество слотов MB — можно добавить >4 модуля в frontend, и только серверная проверка это отловит с задержкой.
**Рекомендация:** Добавить `ramSlots` проверку в `checkBuildCompatibility` (usePCBuilder.ts).

### BUG-12: Проверка Cooler TDP закомментирована на frontend
**Тип:** Неполная реализация
**Место:** `compatibilityUtils.ts:579-586`
**Проблема:** Проверка `cooler.maxTdp < cpu.tdp` закомментирована с комментарием «Not a hard incompatibility but worth noting; not adding to issues to avoid blocking». Результат: пользователь может выбрать кулер, который не справится с TDP процессора, и предупреждение появится только от API (с debounce 120ms).
**Рекомендация:** Сделать хотя бы local Warning (не Error) для TDP несовместимости кулера.

### BUG-13: GPU `recommendedPsu` спецификация игнорируется
**Тип:** Неполная реализация
**Место:** `compatibilityUtils.ts`, `CompatibilityService.cs`
**Проблема:** GPU имеет спецификацию `recommendedPsu` (например, "требуется БП от 750W"), но проверка использует только суммарный TDP + 30% буфер. Если GPU производитель рекомендует 850W, а суммарный TDP расчёт даёт 700W (рекомендуется 910W с буфером), но если производитель GPU говорит "минимум 1000W" — это будет проигнорировано.
**Рекомендация:** Использовать `max(calculated, gpu.recommendedPsu)` для окончательной рекомендации БП.

### BUG-14: `GetCompatibleMotherboardsAsync` и `GetCompatibleRamAsync` — пустые заглушки
**Тип:** Нефункциональный API
**Место:** `CompatibilityService.cs:101-102`
**Проблема:** Эндпоинты возвращают `Enumerable.Empty<Guid>()`. API endpoints существуют в контроллере (строки 334-353), но не работают.
**Рекомендация:** Либо реализовать, либо удалить эндпоинты и вернуть 501 Not Implemented.

## 4. Функциональные ошибки

### BUG-15: `removeComponent('cpu')` удаляет также motherboard, ram, cooling, fan
**Тип:** Потенциально ожидаемое поведение (side effect)
**Место:** `usePCBuilder.ts:743-757`
**Проблема:** При удалении CPU удаляются: motherboard, ram, cooling, fan. Это логично (CPU — основа сборки), но:
- Fan массив очищается полностью, хотя вентиляторы совместимы с любой сборкой
- Cooling очищается, хотя кулеры часто совместимы с новым CPU
Нет обратной связи пользователю о том, что будет очищено при удалении CPU.
**Рекомендация:** Добавить confirmation dialog или оставить только совместимые компоненты. Либо убрать очистку `fan` из логики удаления CPU — вентиляторы не зависят от процессора.

### BUG-16: `removeComponent('motherboard')` удаляет RAM
**Тип:** Потенциально ожидаемое поведение
**Место:** `usePCBuilder.ts:758-759`
**Проблема:** Аналогично BUG-15: `ram: []` очищается при удалении материнской платы. Хотя RAM в общем случае может быть совместима с другой MB — тип DDR должен совпадать, но без MB это невозможно проверить. Частично оправдано.

### BUG-17: `handleChangeQuantity` для удаления использует `-999` для очистки последнего
**Тип:** Хак/небезопасно
**Место:** `ComponentSlot.tsx:144`
**Проблема:** `onChangeQuantity(quantity > 1 ? -1 : -999)` — магическое число `-999` используется как сигнал «полностью очистить слот». В `PCBuilderPage.tsx:212-234` `handleChangeQuantity` обрабатывает `delta < 0` одинаково для любого отрицательного значения, но если логика изменится, `-999` может вызвать неожиданные эффекты.
**Рекомендация:** Явно разделить операции: `remove()` и `clear()`.

### BUG-18: Количество накопителей не отображается в UI (нет quantity control)
**Тип:** Неполный UI
**Место:** `PCBuilderPage.tsx:491-514`
**Проблема:** Для RAM и fan есть `<quantity>` / `onChangeQuantity` пропсы, но для **storage** аналогичные пропсы **НЕ передаются** в `ComponentSlot`. Накопители добавляются через модалку, но нет быстрого +/- управления количеством прямо из слота.
**Рекомендация:** Добавить `quantity`, `maxQuantity`, `onChangeQuantity` в storage слоты аналогично RAM и fan.

### BUG-19: Цена не учитывает промоции
**Тип:** Бизнес-логика
**Место:** `usePCBuilder.ts:562-581`
**Проблема:** Цена рассчитывается как простая сумма `product.price`. Backend имеет эндпоинт `/api/v1/pcbuilder/calculate-price` который «Calculate price with promos via gRPC to CatalogService», но он НЕ используется в frontend.
**Рекомендация:** Вызывать backend для расчёта цены с промоциями, или документировать что промоции не применяются к конструктору.

## 5. API и Backend

### BUG-20: Порт PC Builder service — не 5002, а 5127
**Тип:** Документация/конфигурация
**Место:** `Program.cs:90`, `launchSettings.json:17`
**Проблема:** В ТЗ и curl примерах указан `http://localhost:5002/swagger`, но реальный порт сервиса — `localhost:5127` (или 7248 для HTTPS). Все внешние скрипты, документация и тесты, ссылающиеся на 5002, не будут работать.
**Рекомендация:** Либо изменить порт в launchSettings.json на 5002, либо обновить всю документацию.

### BUG-21: `compatibilityRules.json` дублируется между frontend и backend
**Тип:** Поддерживаемость
**Место:**
- Backend: `src/PCBuilderService/Data/compatibility-rules.json`
- Frontend: `src/frontend/src/config/compatibilityRules.json`
**Проблема:** Один и тот же файл правил существует в двух копиях. Любое изменение правил требует обновления в двух местах. Уже обнаружена разница (дубликат `tr4` в frontend).
**Рекомендация:** Вынести правила в отдельный shared пакет или генерировать frontend JSON из backend-источника в CI.

### BUG-22: API не валидирует `Specifications` Dictionary
**Тип:** Безопасность данных
**Место:** `CompatibilityService.cs:220-222`
**Проблема:** `GetSpecValue` принимает `Dictionary<string, object>?` без строгой типизации. Если frontend отправит некорректные спецификации (например, строку вместо числа для `tdp`), конвертация может вернуть defaults (65W для TDP) вместо ошибки.
**Рекомендация:** Добавить логирование при fallback на default значения, особенно для критичных spec-полей.

## 6. Стили и совместимость браузеров

### BUG-23: CSS `:has()` используеться для thumbnail
**Тип:** Совместимость
**Место:** `ComponentSlot.css:118-123`
**Проблема:** `.component-slot__icon:has(.component-slot__thumbnail)` — псевдокласс `:has()` не поддерживается в Firefox без явного включения (по умолчанию включён с Firefox 121+, но не в ESR).
**Рекомендация:** Добавить fallback через conditional class в React вместо `:has()`.

### BUG-24: Жёстко заданные CSS transition могут конфликтовать с `prefers-reduced-motion`
**Тип:** Доступность
**Место:** `ComponentSlot.css:342-347`
**Проблема:** Медиа-запрос `prefers-reduced-motion: reduce` устанавливает `transition-duration: 0.01ms !important`, но framer-motion анимации (которые выполняются через JS) не отключаются этим CSS правилом.
**Рекомендация:** Проверять `window.matchMedia('(prefers-reduced-motion: reduce)')` в React и передавать `transition={{ duration: 0 }}` в framer-motion.

### BUG-25: Текст спецификаций `font-size: 0.7rem` — на грани читаемости
**Тип:** Доступность/UX
**Место:** `ComponentSlot.css:261`
**Проблема:** `font-size: 0.7rem` = ~10px при стандартном 16px root — ниже минимальных рекомендаций (12px). Особенно проблематично для длинных названий спецификаций.
**Рекомендация:** Поднять до `0.75rem` или `0.8rem`.

## 7. UX/UI проблемы

### BUG-26: Нет подтверждения при удалении ключевых компонентов
**Тип:** UX
**Место:** `ComponentSlot.tsx:206-215`
**Проблема:** Кнопка «Снять» удаляет компонент (и зависимые) мгновенно без подтверждения. Удаление CPU удаляет 5+ компонентов (BUG-15). Пользователь может кликнуть случайно и потерять всю сборку.
**Рекомендация:** Добавить `onBeforeRemove` callback или confirmation toast: «Удаление процессора также удалит материнскую плату, ОЗУ и охлаждение. Продолжить?»

### BUG-27: Периферийные слоты не учитываются в totalPrice при добавлении в корзину
**Тип:** Функциональная ошибка
**Место:** `usePCBuilder.ts:798-822`
**Проблема:** `addToCart` включает fan, monitor, keyboard, mouse, headphones (строки 800-811). Но `totalCount` в хуке = 12 (строка 118: `TOTAL_CATEGORIES = 12`), что включает периферию. При этом `selectedCount` считается на `countSelectedCategories` (строки 493-509), который включает периферию. В целом корректно, но `canAddToCart` требует `isCompatible`, а периферия никаким проверкам не подвергается — неясно должны ли товары с периферией блокировать корзину при несовместимости основных компонентов.
**Рекомендация:** Документировать что периферия не влияет на совместимость, но добавляется в корзину вместе с основными компонентами.

### BUG-28: Build slot order не интуитивен
**Тип:** UX
**Место:** `buildSlotRows()` в `PCBuilderPage.tsx:96-155`
**Проблема:** Порядок слотов: CPU → GPU → MB → RAM → Storage → Fan → PSU → Case → Cooling. **Нет логичной последовательности сборки**. В реальной сборке начинают с MB (процессор + RAM вставляются в плату перед установкой в корпус). Fan стоит между Storage и PSU, что нелогично — вентиляторы обычно последним шагом.
**Рекомендация:** Предложить порядок: CPU → MB → RAM → Storage → Cooling → GPU → Case → PSU → Fan или сделать настраиваемым.

### BUG-29: Нет индикатора загрузки API
**Тип:** UX
**Место:** `usePCBuilder.ts:552` (`isApiLoading` определён, но не используется в UI)
**Проблема:** `isApiLoading` существует и используется в интерфейсе, но нигде не отображается в `PCBuilderPage.tsx` или `BuildSummaryPanel.tsx`. Когда API проверяет совместимость (debounce 120ms), пользователь не видит никакого индикатора.
**Рекомендация:** Добавить спиннер или «Обновление...» рядом со статусом совместимости пока `isApiLoading === true`.

### BUG-30: ComponentPickerModal полностью скрывает несовместимые товары
**Тип:** UX/информация
**Место:** `ComponentPickerModal.tsx:582-584`
**Проблема:** Несовместимые товары скрыты, показывается только «X товаров скрыто». Пользователь не понимает ПОЧЕМУ товар несовместим и не может увидеть какие товары были исключены и по какой причине.
**Рекомендация:** Показать несовместимые товары в полупрозрачном состоянии с иконкой совместимости и tooltip-объяснением.
