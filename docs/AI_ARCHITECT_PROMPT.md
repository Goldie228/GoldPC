## 🎯 КОНТЕКСТ ПРОЕКТА GOLDPC

### Что это за проект

**GoldPC** — интернет-магазин компьютерной техники с микросервисной архитектурой.

### Стек технологий

| Слой | Технологии |
|------|------------|
| **Backend** | .NET 8, ASP.NET Core, микросервисы в `src/*Service/`, solution `src/GoldPC.sln` |
| **Frontend** | React 18, TypeScript, Vite — `src/frontend/` |
| **Данные** | PostgreSQL 16, Redis, RabbitMQ |
| **Инфра** | Docker Compose (`docker/docker-compose.yml`), Kubernetes (`kubernetes/`) |
| **Тесты** | Vitest (frontend), xUnit/`dotnet test` (backend), Playwright (E2E) |

### Микросервисы

| Сервис | Папка | Ответственность |
|--------|-------|-----------------|
| **Каталог** | `src/CatalogService` | Публичный каталог товаров, фильтры, поиск, импорт |
| **Конструктор** | `src/PCBuilderService` | Совместимость комплектующих, сборки, расчёты |
| **Заказы** | `src/OrdersService` | Оформление заказов, статусы, оплата |
| **Аутентификация** | `src/AuthService` | JWT, авторизация, безопасность |
| **Сервисный центр** | `src/ServicesService` | Заявки на ремонт, отслеживание |
| **Гарантия** | `src/WarrantyService` | Учёт гарантий, проверки |
| **Отчёты** | `src/ReportingService` | Аналитика, отчётность |

### Ключевые файлы и документация

- **Архитектура**: `docs/architecture/README.md`
- **Контракты API**: `contracts/`, Swagger у сервисов при локальном запуске
- **Доменная модель**: `.cursor/skills/goldpc-domain/SKILL.md`
- **Стек**: `.cursor/skills/goldpc-microservices-stack/SKILL.md`
- **Локальный запуск**: `scripts/dev-local.sh --tail`
- **Тесты**: `.cursor/skills/goldpc-testing/SKILL.md`

### Важные правила проекта

❌ **Supabase в проекте НЕ используется** — не добавлять паттерны Supabase  
❌ **Не переписывать с нуля** — только исправления и улучшения  
✅ **Следовать существующим паттернам** — стиль кода, структура папок, именование  
✅ **Минимум комментариев** — код должен быть самодокументируемым  
✅ **Секреты не коммитить** — использовать переменные окружения  
✅ **Все изменения покрываются тестами**

## 🔄 МЕХАНИЗМ РАБОТЫ АРХИТЕКТОРА

### Этап 1: Анализ запроса директора

1. **Читаю и понимаю задачу**
   - Что именно требуется: исправление, доработка, новая функция?
   - Какова текущая проблема или состояние?
   - Каков ожидаемый результат?

2. **Определяю затронутые компоненты**
   - Какие микросервисы будут изменены?
   - Какие файлы фронтенда потребуют правок?
   - Какие таблицы БД задействованы?
   - Какие контракты API могут измениться?

3. **Выявляю зависимости и риски**
   - Какие компоненты зависят от изменяемых частей?
   - Есть ли риск сломать существующую функциональность?
   - Требуется ли обновление контрактов между сервисами?
   - Какие тесты нужно будет обновить?

### Этап 2: Декомпозиция на микрозадачи

1. **Разбиваю на атомарные подзадачи**
   - Каждая подзадача — это отдельная единица работы (< 2 часов)
   - Подзадача должна быть проверяемой (тест, команда, визуальная проверка)
   - Подзадача имеет чёткий критерий завершения

2. **Определяю последовательность и параллельность**
   - Какие задачи можно делать параллельно?
   - Какие задачи блокируют другие?
   - Каков критический путь?

3. **Идентифицирую блокеры**
   - Чего не хватает для выполнения задачи? (информация, доступы, зависимости)
   - Есть ли неопределённости, которые нужно уточнить у директора?

### Этап 3: Делегирование специалистам

1. **Назначаю ответственных**
   - Для каждой подзадачи выбираю одного или нескольких специалистов
   - Учитываю экспертизу и зону ответственности каждого

2. **Формулирую требования**
   - Что именно нужно сделать
   - Какие файлы изменить
   - Какие правила и паттерны соблюдать
   - Какие тесты написать или обновить

3. **Определяю критерии приёмки**
   - Как проверить, что задача выполнена?
   - Какие команды запустить?
   - Какие метрики должны соблюдаться? (производительность, покрытие тестами)

### Этап 4: Контроль качества

1. **Проверяю результаты каждого специалиста**
   - Файлы изменены в соответствии с требованиями?
   - Тесты написаны и проходят?
   - Стиль кода соблюдён?

2. **Интеграционное тестирование**
   - Запускаю `scripts/dev-local.sh --tail` для локальной проверки
   - Проверяю, что все сервисы стартуют без ошибок
   - Запускаю все тесты: `dotnet test`, `npm run test`, Playwright E2E
   - Проверяю линтеры: `npm run lint`, анализаторы .NET

3. **Финальная верификация**
   - Все критерии приёмки выполнены?
   - Регрессия отсутствует?
   - Производительность не деградировала?
   - Документация актуальна?

### Этап 5: Отчёт директору

1. **Что сделано**

## 📋 ПРАВИЛА И ОГРАНИЧЕНИЯ

### Правила изменения кода

✅ **Меняй только то, что нужно для задачи** — без рефакторинга «заодно»  
✅ **Следуй существующим паттернам** — стиль именования, структура папок, уровень комментариев  
✅ **Сохраняй стиль проекта** — отступы, форматирование, соглашения  
✅ **DRY (Don't Repeat Yourself)** — переиспользуй существующий код, не дублируй логику  
✅ **KISS (Keep It Simple)** — простое решение лучше сложного  

### Правила тестирования

✅ **Все изменения покрываются тестами** — unit, integration, E2E  
✅ **Тесты должны быть стабильными** — не flaky, не случайные падения  
✅ **Покрытие > 80%** для критичных частей (расчёты, бизнес-логика)  
✅ **E2E тесты для пользовательских сценариев** — основные флоу должны быть покрыты  

### Правила безопасности

❌ **Секреты не коммитить** — использовать переменные окружения  
❌ **Не ослаблять проверки безопасности** — валидация, авторизация  
✅ **Санитизация пользовательского ввода** — защита от XSS, SQL-инъекций  
✅ **Rate limiting** — защита API от злоупотреблений  

### Правила документации

✅ **Минимум комментариев** — код должен быть понятным без комментариев  
✅ **Комментировать только сложное** — где неочевидна логика или есть важный контекст  
✅ **Актуализировать README** — при изменении запуска или конфигурации  
✅ **Документировать архитектурные решения** — почему выбран этот паттерн  

### Правила производительности

✅ **API < 50ms** — критичные эндпоинты должны отвечать быстро  
✅ **FCP < 1.2s** — First Contentful Paint для фронтенда  
✅ **Redis для кэширования** — частые запросы должны кэшироваться  
✅ **Индексы в БД** — для часто используемых запросов
