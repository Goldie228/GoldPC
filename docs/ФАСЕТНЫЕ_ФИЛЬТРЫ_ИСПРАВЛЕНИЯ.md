# Исправления фасетных фильтров - Документация

Этот документ описывает исправления фасетных фильтров, реализованные на основе аудита от 30.03.2026.

## Что было исправлено

### 1. Валидация Range-атрибутов ✅

**Проблема**: Выбросы в диапазонных фильтрах (слайдерах) из-за ошибок парсинга:
- `cases.max_cooler_height`: 1805 мм (должно быть 180.5 мм)
- `mice.dpi`: 32003050 (склеенные числа)
- `monitors.brightness`: 10002 кд/м² (лишние цифры)
- И другие

**Решение**: Создан `SpecificationValidation.cs` с:
- Validation rules для каждого атрибута (мин/макс допустимые значения)
- Специализированные парсеры для разных типов атрибутов
- Автоматическая фильтрация невалидных значений при импорте

**Файлы**:
- `src/CatalogService/Services/SpecificationValidation.cs`
- `src/CatalogService/Services/SpecImportNormalizer.cs` (обновлен)

---

### 2. Percentile-based Aggregation ✅

**Проблема**: Слайдеры отображают выбросы (max 1805 мм вместо реальных 250 мм)

**Решение**: В `ProductRepository.GetSpecificationRangeAsync`:
- Используется 99-й перцентиль вместо голого `Max()`
- Используется 1-й перцентиль вместо голого `Min()`
- Автоматически отсекает верхний и нижний 1% выбросов

**Файлы**:
- `src/CatalogService/Repositories/ProductRepository.cs`

---

### 3. Фильтрация Leaked Values ✅

**Проблема**: В Terms-фильтры попадают значения из других полей:
- `keyboards.type`: батарейки (AA, AAA), свитчи (Razer Green)
- `mice.type`: батарейки, интерфейс (Bluetooth)
- `monitors.type`: яркость (200 кд/м2)
- `headphones.type`: крепления микрофона
- `coolers.type`: коннекторы питания (2 pin, 4 pin)

**Решение**: В `SpecImportNormalizer.NormalizeForLookup`:
- Автоматическая фильтрация leaked values через regex
- Значения пропускаются при индексации (возвращается пустая строка)
- Логирование пропущенных значений

**Файлы**:
- `src/CatalogService/Services/SpecImportNormalizer.cs`

---

### 4. Нормализация дубликатов ✅

**Проблема**: Дубликаты Terms-значений:
- `motherboards.form_factor`: `Mini-ITX` и `miniITX`
- `motherboards.chipset`: `B650` и `AMD B650`
- `processors.model_series`: `Core i7` и `Intel Core i7`
- `processors.memory_support`: `DDR4, DDR5` и `DDR5, DDR4` (разный порядок)
- `storage.interface`: двойные пробелы, оксюмороны (SATA + NVMe)

**Решение**: В `SpecImportNormalizer.NormalizeForLookup`:
- Унификация форм-факторов (приведение к единому регистру)
- Сортировка частей в memory_support
- Удаление лишних пробелов
- Удаление оксюморонов (SATA 3.0 (NVMe) → SATA 3.0)

**Файлы**:
- `src/CatalogService/Services/SpecImportNormalizer.cs`

---

### 5. Нормализация словарей ✅

**Проблема**: Смешивание локализаций и технических терминов:
- `psu.efficiency`: 17 вариантов (`false`, `80+ Gold`, `золотой`, `94 %`)
- `psu.modular`: 8 вариантов (`false`, `Full`, `полностью модульное`)

**Решение**: В `SpecImportNormalizer.NormalizeForLookup`:
- `psu.efficiency`: маппинг всех вариантов → единый формат ("80+ Gold (золотой)")
- `psu.modular`: маппинг → 3 варианта (Немодульный, Полумодульный, Полностью модульный)
- Аналогично для других атрибутов

**Файлы**:
- `src/CatalogService/Services/SpecImportNormalizer.cs`

---

### 6. Конвертация единиц измерения ✅

**Проблема**: Неправильная конвертация МГц ↔ ГГц:
- `processors.base_freq`: 1.5 – 4200 ГГц (макс в МГц)
- `processors.max_freq`: 3 – 5300 ГГц (макс в МГц)

**Решение**: В `SpecificationValidation.ParseFrequency`:
- Автоматическая конвертация: если значение > 100, делим на 1000 (МГц → ГГц)
- Валидация: базовая частота 0.5-6.0 ГГц, макс 1.0-6.5 ГГц

**Файлы**:
- `src/CatalogService/Services/SpecificationValidation.cs`

## Миграция существующих данных

### Доступные миграции

Создан сервис `SpecificationDataMigration` с тремя миграциями:

#### 1. FixRangeOutliersAsync
Исправляет выбросы в Range-атрибутах:
- Пытается исправить (1805 → 180, 32003050 → 3200)
- Удаляет невалидные значения, которые не удалось исправить
- Логирует все изменения

#### 2. RemoveLeakedValuesAsync
Удаляет leaked values из Terms-атрибутов:
- Батарейки из `type` (клавиатуры/мыши)
- Свитчи из `keyboards.type`
- Яркость из `monitors.type`
- Крепления микрофона из `headphones.type`
- Коннекторы из `coolers.type`
- Булево "false" из `headphones.type`

#### 3. NormalizeDuplicatesAsync
Объединяет дубликаты в Terms-атрибутах:
- Форм-факторы (Mini-ITX vs miniITX)
- Чипсеты (B650 vs AMD B650)
- Поддержка памяти (разный порядок)
- Интерфейсы накопителей (двойные пробелы, оксюмороны)

### Как запустить миграции

#### Вариант 1: Через API (рекомендуется)

**Все миграции сразу** (последовательно):
```bash
curl -X POST http://localhost:5000/api/v1/admin/data/migrate/all \
  -H "Authorization: Bearer YOUR_ADMIN_TOKEN"
```

**По отдельности**:
```bash

```

## Автоматические исправления при импорте

Все новые товары автоматически проходят через:

1. **Валидацию Range-значений**:
   - Невалидные значения логируются и пропускаются
   - В логах: `"Validation failed for attribute {Key} with value {Value}"`

2. **Фильтрацию Leaked Values**:
   - Значения пропускаются автоматически
   - В логах: `"Skipping leaked value '{Value}' for attribute '{Key}'"`

3. **Нормализацию Terms-значений**:
   - Автоматическое приведение к единому формату
   - Объединение дубликатов

4. **Конвертацию единиц**:
   - Автоматическая конвертация МГц → ГГц для частот
   - Безопасный парсинг DPI, яркости, размеров

## Мониторинг и логирование

### Логи валидации

Все validation failures логируются с уровнем `Warning`:

```log
[10:15:23 WRN] Validation failed for attribute max_cooler_height with value 1805 on product 12345-67890: Значение 1805 мм выходит за допустимый диапазон 0-300. Максимальная высота кулера в реальных корпусах ~250 мм
```

### Логи миграции

Миграции логируются с уровнем `Information`:

```log
[10:20:15 INF] Starting range outliers fix migration
[10:20:15 INF] Processing 5432 values for attribute max_cooler_height
[10:20:16 INF] Fixed outlier for max_cooler_height: 1805 → 180 (Product: abc123)
[10:20:17 WRN] Removed invalid value for dpi: 32003050 (Product: def456). Reason: Значение 32003050 DPI выходит за допустимый диапазон 100-100000
[10:20:25 INF] Range outliers fix completed. Valid: 5200, Fixed: 180, Removed: 52
```

### Мониторинг через метрики

Рекомендуется добавить метрики для отслеживания:
- Количество отброшенных значений по каждому атрибуту
- Процент validation failures
- Алерты при > 5% failures

## FAQ

### Q: Что делать, если legitimate значения отбрасываются?

A: Проверить validation rules в `SpecificationValidation.cs` и скорректировать границы. Например:
```csharp
["max_cooler_height"] = new ValidationRule
{
    MinValue = 0,
    MaxValue = 350, // Увеличить, если есть корпуса с высокими кулерами
    Unit = "мм"
}
```

### Q: Как добавить новый атрибут с валидацией?

A: Добавить правило в `SpecificationValidation.ValidationRules`:
```csharp
["new_attribute"] = new ValidationRule
{
    MinValue = 10,
    MaxValue = 1000,
    Unit = "единица",
    Reason = "Объяснение границ"
}
```

### Q: Как добавить нормализацию для нового Terms-атрибута?

A: Добавить case в `SpecImportNormalizer.NormalizeForLookup`:
```csharp
if (string.Equals(attrKey, "new_attribute", StringComparison.OrdinalIgnoreCase))
{
    if (normalized == "variant1" || normalized == "alias1") 
        return "canonical_value";
}
```

### Q: Нужно ли запускать миграции после каждого импорта?

A: **Нет**. Миграции нужны только **один раз** для исправления существующих данных. Все новые товары автоматически проходят через валидацию и нормализацию при импорте.
