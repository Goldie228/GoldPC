## 🎯 Что реализовано

### 1. Система валидации Range-атрибутов

**Файл**: `src/CatalogService/Services/SpecificationValidation.cs`

**Что делает**:
- Validation rules для каждого атрибута (min/max границы)
- Специализированные парсеры для разных типов данных
- Автоматическая конвертация единиц (МГц→ГГц, МБ→ГБ)
- Фильтрация физически невозможных значений

**Примеры правил**:
```csharp
["max_cooler_height"] = new ValidationRule
{
    MinValue = 0,
    MaxValue = 300,  // Реальный макс ~250 мм
    Unit = "мм",
    Reason = "Максимальная высота кулера в реальных корпусах ~250 мм"
}

["dpi"] = new ValidationRule
{
    MinValue = 100,
    MaxValue = 100000,  // Отсекает склеенные 32003050
    Unit = "DPI"
}
```

**Специализированные парсеры**:
- `ParseDpi`: Разбивает склеенные числа (32003050 → 3200)
- `ParseBrightness`: Исправляет лишние цифры (10002 → 1000)
- `ParseFrequency`: Конвертирует МГц → ГГц (4200 → 4.2)
- `ParseVideoMemory`: Конвертирует МБ → ГБ (512 → 0.5)
- `ParseFanSize`: Извлекает диаметр из "1x140 мм" (→ 140)

### 2. Percentile-based Aggregation

**Файл**: `src/CatalogService/Repositories/ProductRepository.cs`  
**Метод**: `GetSpecificationRangeAsync` (строки 406-489)

**Что изменено**:
```csharp
// БЫЛО:
return (nums.Min(), nums.Max());

// СТАЛО:
var sortedNums = nums.OrderBy(n => n).ToList();
var p1Index = (int)Math.Floor(count * 0.01);   // 1-й перцентиль
var p99Index = (int)Math.Ceiling(count * 0.99) - 1;  // 99-й перцентиль

min = sortedNums[p1Index];
max = sortedNums[p99Index];
```

**Эффект**: Автоматически отсекает верхний и нижний 1% выбросов при формировании слайдеров.

### 3. Фильтрация Leaked Values

**Файл**: `src/CatalogService/Services/SpecImportNormalizer.cs`  
**Метод**: `ShouldSkipValue` (новый)

**Что фильтруется**:

| Атрибут | Leaked Values | Regex Pattern |
|---------|---------------|---------------|
| `keyboards.type` | Батарейки | `(\d+\s*x\s*)?(AA|AAA|Li-\w+)` |
| `keyboards.type` | Свитчи | `(Razer|Kailh|Outemu)\s+(Red|Blue|Green)` |
| `mice.type` | Батарейки + интерфейс | `AA|AAA|Li-\w+|Bluetooth|USB` |
| `monitors.type` | Яркость | `\d+\s*(кд/м|cd/m|nits)` |
| `cases.form_factor` | Физ. размеры | `\d+["']?\s*x\s*\d+|до\s+\d+\s*мм` |
| `headphones.type` | Крепления микрофона | `держатель|встроенный в корпус` |
| `coolers.type` | Коннекторы питания | `\d+\s*pin|Molex|SATA|USB` |

**Как работает**:
1. При импорте товара значение проверяется через `ShouldSkipValue`
2. Если совпадает с паттерном — возвращается `string.Empty`
3. Значение пропускается (не добавляется в индекс)
4. Логируется: `"Skipping leaked value 'AA' for attribute 'type'"`

### 5. Миграционный сервис

**Файл**: `src/CatalogService/Services/SpecificationDataMigration.cs`

**API Endpoints** (требуют Admin-роль):
- `POST /api/v1/admin/data/migrate/fix-range-outliers`
- `POST /api/v1/admin/data/migrate/remove-leaked-values`
- `POST /api/v1/admin/data/migrate/normalize-duplicates`
- `POST /api/v1/admin/data/migrate/all` (все сразу)

**Что делает**:

#### FixRangeOutliersAsync
1. Находит все невалидные Range-значения
2. Пытается исправить:
   - `1805` → `180` (потеряна точка)
   - `3200` → `320` (потеряна точка)
   - `32003050` → `3200` (склеенные числа)
   - `10002` → `1000` (лишние цифры)
3. Если не удалось — удаляет
4. Возвращает статистику: Valid / Fixed / Removed

#### RemoveLeakedValuesAsync
1. Находит все leaked canonical values (по regex паттернам)
2. Удаляет canonical value + все связанные product values
3. Логирует каждое удаление

#### NormalizeDuplicatesAsync
1. Объединяет дубликаты:
   - Форм-факторы (регистр/дефисы)
   - Чипсеты (префикс производителя)
   - Поддержка памяти (порядок)
   - Интерфейсы (пробелы/оксюмороны)
2. Перенаправляет все product values на canonical value
3. Удаляет дубликаты

## 🔧 Технические детали реализации

### Архитектура решения

```
Импорт товара
    ↓
SpecImportNormalizer.ToSpecificationValuesAsync
    ↓
    ├─→ Range-атрибут
    │   ├─→ SpecificationValidation.ParseAndValidateNumber
    │   │   ├─→ ParseDpi / ParseBrightness / ParseFrequency / ...
    │   │   └─→ ValidateNumber (check min/max)
    │   └─→ Логирование validation failures
    │
    └─→ Select-атрибут
        ├─→ NormalizeForLookup
        │   ├─→ ShouldSkipValue (фильтрация leaked values)
        │   └─→ Словари нормализации (efficiency, modular, etc.)
        └─→ Поиск/создание CanonicalValue
```

### Обработка на read-path (фасеты)

```
ProductRepository.GetSpecificationRangeAsync
    ↓
Получить все ValueNumber для атрибута
    ↓
Percentile-based aggregation (1% и 99%)
    ↓
Вернуть (min, max) без выбросов
```

## ⚠️ Риски и откаты

### Риски

1. **Потеря legitimate данных**: Если validation boundaries слишком строгие
   - Митigation: Логировать все отброшенные значения, проверить перед миграцией
   
2. **Performance impact**: Percentile aggregation медленнее голого Min/Max
   - Митigation: Только для больших выборок (>10 значений), добавить кэширование

3. **Breaking changes для существующих фильтров**: Клиенты могут полагаться на старые значения
   - Mitigation: Версионирование API, graceful degradation

## 🎓 Что полезно изучить, Goldie

Поскольку ты изучаешь backend, вот ключевые концепции из этой задачи:

### 1. Валидация данных на уровне доменной логики
- **Domain-Driven Design**: Validation rules как часть domain model
- **Fail-fast principle**: Отбрасываем невалидные данные при импорте, а не при чтении
- **Defensive programming**: Не доверяем сырым данным из внешних источников

### 2. Data normalization patterns
- **Dictionary/mapping tables**: raw value → canonical value
- **Canonical values**: Единый источник истины для Terms-атрибутов
- **Denormalization for performance**: AttributeKey дублируется в CategoryFilterAttribute

### 3. Percentile-based aggregation
- **Outlier detection**: 99-й перцентиль вместо Max() для устойчивости к выбросам
- **Когда применять**: Большие выборки с возможными ошибками данных
- **Trade-off**: Точность vs робастность (теряем верхний 1%, но слайдеры работают)

### 4. Migration patterns
- **Idempotency**: Миграция должна быть безопасна для повторного запуска
- **Transaction safety**: Весь батч в одной транзакции или rollback
- **Logging and monitoring**: Каждое изменение логируется для audit trail

### 5. API design для admin operations
- **Authorization**: Миграции только для Admin-роли
- **Observability**: Возврат детальной статистики (Valid/Fixed/Removed)
- **Safety**: POST endpoints, не GET (изменяют состояние)
