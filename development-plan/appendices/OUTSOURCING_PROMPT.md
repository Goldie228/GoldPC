## Текущее состояние кодовой базы

Перед выполнением задач запустите `./scripts/dev-local.sh --tail` и убедитесь, что проект собирается и запускается. Изучите реальный код и скрипты — не полагайтесь на README или docs/, они могут быть устаревшими.

### Что уже реализовано (production-ready):
- **CatalogService** (port 5000): полный CRUD товаров, категории, фильтры, спецификации, загрузка изображений, seed из X-Core, RabbitMQ consumers, gRPC endpoint
- **AuthService** (port 5001): регистрация, login/logout, JWT + refresh tokens, 6 ролей, wishlist, адреса пользователей
- **OrdersService** (port 5002): создание заказов, корзина, checkout, promocodes, outbox pattern, gRPC-клиент к CatalogService, mock Stripe, 1C webhook (stub)
- **PCBuilderService** (port 5005): сохранение/загрузка конфигураций, rule-based совместимость (socket, DDR, PSU, form-factor), FPS estimation
- **Frontend**: все страницы каталога, товара, конструктора, корзины, checkout, авторизации, аккаунта, админки, услуг, гарантии

### Что реализовано частично (mock/stub):
- Платёжная интеграция (Stripe mock, не подключена ЮKassa)
- SMS/email уведомления (mock-реализации)
- Резервирование stock (stub в OrdersService)
- ServicesService — базовый CRUD, не enforced ограничение мастера ≤ 3 активных заявок
- WarrantyService — базовое создание гарантий, сложные workflow не реализованы
- 1C export — controller есть, интеграция stub

## Техническое задание (ТЗ)

Полное ТЗ с требованиями находится в файле `development-plan/appendices/ТЗ_GoldPC.md`. Все требования имеют уникальные коды (ФТ-*, НФТ-*, ПТ-*, UC-*, БТ-*).

### Ключевые модули и требования из ТЗ:

**Модуль 1: Каталог товаров (ФТ-1.1 — ФТ-1.10)**
- Просмотр без авторизации, фильтрация по категориям/производителю/цене
- Поиск, изображения, рейтинг, отзывы, остатки на складе

**Модуль 2: Конструктор ПК (ФТ-2.1 — ФТ-2.11)**
- Интерактивный подбор с автоматической фильтрацией по сокету, DDR, PSU
- Проверка баланса CPU-GPU, мощности БП, форм-фактора корпуса
- Сохранение конфигурации, добавление в корзину

**Модуль 3: Заказы (ФТ-3.1 — ФТ-3.13)**
- Корзина → checkout, онлайн/при получении, самовывоз/доставка
- Резервирование, статусы, email/SMS уведомления, история изменений
- Лимит — 5 единиц одного товара, promo-коды, скидки

**Модуль 4: Услуги (ФТ-4.1 — ФТ-4.11)**
- Диагностика, ремонт, сборка ПК, чистка, профилактика
- Заявки со статусами, назначение мастера (≤ 3 активных)
- Учёт работ и запчастей, расчёт стоимости

**Модуль 5: Гарантия (ФТ-5.1 — ФТ-5.9)**
- Авто-создание при продаже, 14 дней на услуги, 12-36 мес на комплектующие
- Статусы: действует/истекла/аннулирована, уведомления за месяц

**Модуль 6: Администрирование (ФТ-6.1 — ФТ-6.9)**
- 6 ролей: Гость, Клиент, Менеджер, Мастер, Администратор, Бухгалтер
- CRUD пользователей, справочники, audit log, системные настройки

## Правила работы с кодовой базой

### Backend (C# / ASP.NET Core):
- EF Core 8, FluentValidation, Serilog
- Все сервисы используют общий `SharedKernel` (DTO, entities, enums)
- Межсервисная коммуникация: REST (синхронная) + RabbitMQ (асинхронная)
- gRPC для высокопроизводительных запросов (CatalogService)
- Outbox pattern для надёжной отправки сообщений
- Миграции через EF Core Migrations
- Логирование через Serilog с request ID

### Frontend (React / TypeScript):
- Zustand для state management (не Redux)
- Custom CSS modules (не Material-UI, это устаревшее требование в ТЗ)
- MSW для моков в dev-режиме
- API-клиенты в `src/frontend/src/api/`
- Хуки для бизнес-логики (usePCBuilder, useCompatibility, etc.)
- Компоненты: Button, Input, Card, Modal, Toast, Skeleton, RangeSlider, Tabs

### Инфраструктура:
- Docker Compose: Postgres primary+replica, Redis, RabbitMQ, CatalogService, AuthService, Frontend, Adminer
- Остальные сервисы запускаются через `scripts/dev-local.sh`
- Helm charts для K8s, Terraform для IaC (базовые, не полностью функциональные)
- Prometheus + Grafana для мониторинга

### Тестирование:
- Backend: unit-тесты xUnit, интеграционные тесты с Testcontainers, Pact contract tests, ArchitectureTests
- Frontend: компонентные тесты (Vitest), hook-тесты
- E2E: Playwright с Page Object Model, BDD с Cucumber
- Целевое покрытие: ≥ 70% (ТЗ, раздел 5.2.3)

## Критерии приёмки (общие)

Каждая задача должна:

1. **Собирать** — `dotnet build` и `npm run build` без ошибок
2. **Проходить существующие тесты** — `dotnet test` и `npm test` все проходят
3. **Покрытие тестами** — новые/изменённые функции покрыты unit-тестами
4. **Следовать стилю** — существующие naming convention, архитектура, паттерны
5. **Безопасность** — валидация входных данных (FluentValidation), параметризованные запросы (EF Core), проверка авторизации на endpoint-ах
6. **Миграции** — изменения схемы БД сопровождаются EF Core migration
7. **Документация** — обновлены Swagger-аннотации для изменившихся endpoint-ов

```
1. Создать ветку: git checkout -b feature/<task-id>-<short-description>
2. Реализовать изменения
3. Запустить тесты: dotnet test && npm test

5. Проверить вручную новый функционал

7. Создать PR в основную ветку
```

## Известные проблемы и ограничения

1. README и docs/ могут быть устаревшими — всегда проверяй реальный код и `scripts/dev-local.sh`
2. OrdersService, PCBuilderService, ServicesService, WarrantyService НЕ включены в базовый docker-compose — запускаются через dev-local.sh
3. ЮKassa не подключена — используется Stripe mock в dev
4. Покрытие тестами < 70% — это не блок, но новые тесты обязательны для новых функций
5. Kubernetes manifests и Terraform — заглушки, не готовы для production
