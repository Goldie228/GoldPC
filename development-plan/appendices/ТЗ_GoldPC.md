## 1. ВВЕДЕНИЕ
### 1.1. Наименование и область применения
1.1.1.Настоящее техническое задание (ТЗ) определяет требования к разработке веб-приложения «GoldPC» — информационной системы для автоматизации деятельности компьютерного магазина с сервисным центром.
1.1.2.Область применения: розничная торговля компьютерными комплектующими, сборка персональных компьютеров под заказ, оказание услуг по ремонту и обслуживанию компьютерной техники.
1.1.3.Система предназначена для использования в малых и средних предприятиях розничной торговли и сервисных центрах с числом пользователей до 50 человек.
### 1.2. Назначение разработки
#### 1.2.1. Функциональное назначение
1.2.1.1.Веб-приложение «GoldPC» предназначено для автоматизации следующих бизнес-процессов:
- Управление каталогом товаров с фильтрацией по техническим характеристикам
- Онлайн-конструктор ПК с автоматической проверкой совместимости комплектующих
- Оформление и обработка заказов на товары и услуги
- Управление складским учётом и резервированием товаров
- Учёт гарантийных случаев и обслуживания
- Управление заявками на ремонт и сборку
- Администрирование пользователей и прав доступа
- Формирование отчётов и аналитика
1.2.1.2.Система поддерживает 6 ролей пользователей:
- Гость (неавторизованный пользователь)
- Клиент (зарегистрированный покупатель)
- Менеджер (обработка заказов и управление товарами)
- Мастер (выполнение ремонтных и сборочных работ)
- Администратор (управление системой)
- Бухгалтер (финансовые отчёты)
#### 1.2.2. Эксплуатационное назначение
1.2.2.1.Система должна обеспечивать:
- Доступность 99.5% в год (не более 3.65 часов простоя)
- Время отклика на действия пользователя не более 2 секунд
- Одновременную работу до 100 пользователей
- Хранение данных без ограничения срока (архивирование по необходимости)
- Резервное копирование базы данных ежедневно
1.2.2.2.Эксплуатация системы предполагает:
- Размещение на сервере с ОС Linux (Ubuntu 20.04+)
- Использование PostgreSQL 16 в качестве основной СУБД
- Использование Redis 7 для кэширования часто используемых данных
- Поддержка браузеров: Chrome 90+, Firefox 88+, Safari 14+, Edge 90+
- Развёртывание в контейнерах Docker

## 2. ТРЕБОВАНИЯ К ПРОГРАММЕ
### 2.1. Требования к функциональным характеристикам
#### 2.1.1. Состав функций
2.1.1.1.Система должна реализовывать следующие модули и функции:
Модуль 1: Каталог товаров (ФТ-1.1 — ФТ-1.10)- Просмотр каталога без авторизации
- Фильтрация по категориям (процессоры, видеокарты, материнские платы, ОЗУ, БП, накопители, корпуса, периферия)
- Фильтрация по производителю и цене (диапазон)
- Отображение технических характеристик (сокет, частота, объём, тип разъёма, мощность, форм-фактор)
- Показ актуального остатка на складе
- Отображение цены и гарантийного срока
- Поиск по названию и описанию
- Отображение изображений, рейтинга и отзывов
Модуль 2: Конструктор ПК (ФТ-2.1 — ФТ-2.11)- Интерактивный конструктор с выбором назначения (игровой, офисный, рабочая станция)
- Последовательный подбор компонентов с автоматической фильтрацией:
  - Выбор процессора → фильтрация материнских плат по сокету
  - Выбор материнской платы → фильтрация ОЗУ по типу (DDR4/DDR5) и частоте
  - Выбор видеокарты, блока питания, корпуса, накопителя
- Проверка баланса производительности (процессор-видеокарта)
- Проверка достаточности мощности блока питания
- Проверка совместимости корпуса (форм-фактор, длина видеокарты)
- Отображение предупреждений при несовместимости
- Расчёт итоговой стоимости
- Сохранение конфигурации (для авторизованных)
- Добавление конфигурации в корзину с услугой сборки
Модуль 3: Заказы (ФТ-3.1 — ФТ-3.13)- Формирование заказа из корзины
- Два способа оплаты: онлайн и при получении
- Два способа получения: самовывоз и доставка
- Проверка наличия товаров на складе при оформлении
- Резервирование товаров при подтверждении
- Автоматическое присвоение уникального номера заказа
- Статусы заказа: Новый, В обработке, Оплачен, Передан в сборку/ремонт, Готов, Выдан, Отменён
- Изменение статуса менеджером
- Email/SMS-уведомления клиента об изменении статуса
- Просмотр статуса заказа в личном кабинете
- Ограничение количества одного товара до 5 единиц
- Ведение истории изменений статусов
- Расчёт общей стоимости с учётом скидок
Модуль 4: Услуги (ФТ-4.1 — ФТ-4.11)- Виды услуг: диагностика, ремонт, сборка ПК, чистка, профилактика
- Оформление заявки на услугу клиентом
- Назначение уникального идентификатора заявки
- Статусы услуги: Принята, В работе, Выполнена, Выдана
- Назначение мастера на заявку
- Ограничение активных заявок у мастера до 3
- Изменение статуса мастером
- Внесение данных о выполненных работах и использованных запчастях
- Расчёт стоимости услуги (работы + запчасти)
- Уведомление клиента о готовности
- Ведение истории изменений статусов
Модуль 5: Гарантия (ФТ-5.1 — ФТ-5.9)- Автоматическое создание гарантийного талона при продаже товара
- Создание гарантии на услуги сроком на 14 дней
- Гарантийный срок на комплектующие от производителя (12-36 месяцев)
- Статусы гарантии: Действует, Истекла, Аннулирована
- Проверка срока действия при обращении
- Фиксация всех операций по гарантийному обслуживанию
- Возможность аннулирования гарантии с указанием причины
- Генерация уведомлений о приближении окончания гарантии (за месяц)
- Учёт гарантийных случаев с фильтрацией
Модуль 6: Администрирование (ФТ-6.1 — ФТ-6.9)- Ролевая модель доступа: Гость, Клиент, Менеджер, Мастер, Администратор, Бухгалтер
- Управление учётными записями сотрудников (создание, редактирование, блокировка)
- Сброс паролей пользователей
- Аутентификация по логину/паролю
- Авторизация на основе ролей
- Управление справочниками (категории товаров, виды услуг, типы устройств)
- Ведение журнала аудита критических операций
- Настройка параметров системы (лимиты, уведомления)
- Полный доступ ко всем данным системы для администратора
#### 2.1.2. Организация входных и выходных данных
2.1.2.1. Входные данные:- Данные каталога товаров (название, категория, производитель, характеристики, цена, остаток)
- Данные пользователей (логин, хэш пароля, роль, персональные данные)
- Данные заказов (состав, адрес, способ оплаты/получения, контактные данные)
- Данные заявок на ремонт (описание проблемы, модель устройства, серийный номер, фото)
- Данные о статусах и их изменениях
- Данные о гарантийных талонах
2.1.2.2. Выходные данные:- Страницы каталога с фильтрами и пагинацией
- Страницы товаров с детальными характеристиками
- Конфигурации ПК из конструктора
- Документы заказов (счета, накладные)
- Уведомления (email, SMS)
- Отчёты (продажи, статистика, финансы)
- Гарантийные талоны
- Журналы аудита
2.1.2.3.Все входные данные должны проходить валидацию на стороне сервера. Выходные данные должны быть экранированы для защиты от XSS-атак.

### 2.2. Требования к надёжности
#### 2.2.1. Требования к отказоустойчивости
2.2.1.1.Система должна обеспечивать доступность (uptime) не менее 99.5% в год (НФТ-2.1), что соответствует не более 3.65 часов простоя annually.
2.2.1.2.Система должна быть отказоустойчива к временной недоступности Redis (НФТ-2.3):
- При недоступности Redis система должна автоматически переключаться на кэширование в памяти приложения
- После восстановления Redis кэш должен синхронизироваться
- Отсутствие данных в кэше не должно приводить к сбоям в работе
2.2.1.3.Система должна продолжать функционировать при временной недоступности внешних сервисов (email, SMS):
- Очереди задач для асинхронной отправки уведомлений
- Повторные попытки отправки (retry mechanism)
- Логирование неудачных попыток
2.2.1.4.При сбое в работе одного веб-сервера нагрузка должна автоматически перераспределяться на другие экземпляры (load balancing).
2.2.1.5.Система должна корректно обрабатывать ошибки подключения к базе данных:
- Повторные попытки подключения (connection retry)
- Graceful degradation (отображение сообщения о временной недоступности)
- Логирование инцидентов
#### 2.2.2. Требования к восстанавливаемости
2.2.2.1.Время восстановления после сбоя (НФТ-2.2) должно составлять не более 30 минут, включая:
- Перезапуск сервисов (≤5 минут)
- Восстановление из резервной копии базы данных (≤20 минут)
- Проверка целостности данных (≤5 минут)
2.2.2.2.Резервное копирование базы данных (НФТ-2.4):
- Ежедневное полное резервное копирование в 02:00
- Хранение резервных копий не менее 30 дней
- Возможность восстановления на любой момент времени (point-in-time recovery)
- Автоматическая проверка целостности резервных копий
2.2.2.3.Хранение истории изменений (НФТ-2.5):
- История изменений заказов и гарантийных талонов хранится неограниченно
- Удаление записей из основной таблицы не должно приводить к потере истории
- Возможность аудита всех изменений через журнал аудита
2.2.2.4.Система должна вести журнал аудита (audit log) всех критических операций:
- Изменение статуса заказа
- Изменение данных пользователя
- Изменение цен и остатков товаров
- Аннулирование гарантии
- Вход/выход пользователей с повышенными привилегиями

2.7.1.Для веб-приложения требования к транспортированию и хранению адаптируются следующим образом:
2.7.2. Транспортирование:-Доставка кода:через Git (GitHub, GitLab, Bitbucket)
-Доставка артефактов:Docker-образы через Registry
-Требования к безопасности:использование HTTPS для Git и Registry, аутентификация через SSH-ключи или токены
-Целостность:проверка хэш-сумм (Docker делает это автоматически)
2.7.3. Хранение:-Исходный код:в системе контроля версий (Git) с регулярными коммитами
-Docker-образы:в Registry с тегированием по версиям
-Резервные копии базы данных:на отдельном хранилище (S3, NAS, другой сервер) с шифрованием
-Логи:централизованное хранение (ELK Stack, Loki) или ротация на диске (logrotate)
-Конфиденциальные данные:(пароли, ключи API) хранятся в vault (HashiCorp Vault, Azure Key Vault) или как environment variables с шифрованием
2.7.4. Сроки хранения:-Исходный код:бессрочно
-Docker-образы:минимум 5 последних версий, остальные можно удалять
-Резервные копии БД:30 дней (ежедневные) + 12 ежемесячных (полных на 1-е число)
-Логи:90 дней (возможно меньше для debug-логов, 365 дней для audit-логов)
2.7.5. Условия хранения:-Git-репозиторий:на защищённом сервере с бэкапами
-Docker Registry:с аутентификацией и авторизацией
-Резервные копии:на отдельном физическом носителе (не на том же сервере, что и основная БД)
-Логи:с контролем размера (ротация), защита от несанкционированного доступа

#### 2.8.2. Требования к безопасности
2.8.2.1. Аутентификация:- Обязательная аутентификация для всех действий кроме просмотра каталога (НФТ-3.6)
- Хэширование паролей с солью (bcrypt с cost factor 12 или Argon2id)
- JWT-токены с сроком жизни 15 минут (access token) и 7 дней (refresh token)
- Возможность смены пароля
- Восстановление пароля через email (токен с ограниченным сроком)
2.8.2.2. Авторизация:- Проверка прав доступа на каждый запрос (НФТ-3.7)
- Политики на уровне контроллеров и действий:
  - Только администратор может управлять пользователями
  - Только менеджер может изменять статусы заказов
  - Только мастер может обновлять заявки на ремонт
  - Клиент видит только свои заказы
- Запрет горизонтального повышения привилегий (IDOR-уязвимости)
2.8.2.3. Аудит (НФТ-3.8):- Логирование всех критических операций:
  - Вход/выход пользователя (с IP-адресом)
  - Изменение данных пользователя
  - Изменение статуса заказа/услуги
  - Изменение цен и остатков товаров
  - Создание/аннулирование гарантии
  - Сброс пароля
- Хранение логов не менее 365 дней
- Защита логов от изменения (immutable logs)
2.8.2.4. Защита от распространённых атак:-SQL-инъекции (НФТ-3.3):только параметризованные запросы, запрет конкатенации строк
-XSS (НФТ-3.4):экранирование вывода, CSP-заголовки (Content Security Policy)
-CSRF (НФТ-3.5):CSRF-токены для всех state-changing операций
-Clickjacking:X-Frame-Options: DENY
-MIME-sniffing:X-Content-Type-Options: nosniff
-HSTS:Strict-Transport-Security для принудительного HTTPS
2.8.2.5. Шифрование данных:- HTTPS (TLS 1.2+) для всего трафика (НФТ-3.2)
- Шифрование конфиденциальных полей в БД (телефон, email, адрес) (НФТ-3.9)
- Использование современных шифров (AES-256-GCM)
- Ключи шифрования хранятся в vault или как environment variables
2.8.2.6. Мониторинг и реагирование:- Логирование всех попыток неудачного входа
- Оповещение администратора о подозрительной активности (много неудачных попыток)
- Регулярный аудит безопасности (сканирование уязвимостей)
- Обновление зависимостей (dependencies) с исправлениями безопасности
#### 2.8.3. Требования к документации

## 6. ПОРЯДОК КОНТРОЛЯ И ПРИЁМКИ
### 6.1. Виды испытаний
6.1.1.Для обеспечения качества системы проводятся следующие виды испытаний:
#### 6.1.1.1. Функциональное тестирование
- Цель:проверка соответствия функциональных требований (раздел 2.1)
- Метод:black-box тестирование, тест-дизайн (эквивалентное разбиение, анализ граничных значений)
- Объём:все use cases из отчёта ИТК3 (UC-01 — UC-06) и все функциональные требования (ФТ-1.1 — ФТ-6.9)
- Критерий успеха:100% прохождение позитивных сценариев, ≤5% известных багов (не критических)
#### 6.1.1.2. Нагрузочное тестирование
- Цель:проверка требований к производительности (НФТ-1.1 — НФТ-1.5)
- Сценарии:
- 100 одновременных пользователей (нормальная нагрузка)
- 200 одновременных пользователей (пиковая нагрузка)
- 500 одновременных пользователей (стресс-тест)
- Метрики:
- Время отклика (response time) ≤2 секунды для 95% запросов
- Пропускная способность (throughput) ≥50 запросов/секунду
- Ошибки (error rate) ≤1%
- Инструменты:JMeter, k6, Locust
#### 6.1.1.3. Тестирование надёжности
- Цель:проверка требований к надёжности (раздел 2.2)
- Сценарии:
- Доступность системы в течение 72 часов (проверка uptime ≥99.5%)
- Восстановление после сбоя (перезапуск сервисов, восстановление из бэкапа)
- Отказоустойчивость к недоступности Redis (система продолжает работать)
- Критерий:время восстановления ≤30 минут
#### 6.1.1.4. Тестирование безопасности
- Цель:проверка требований к безопасности (раздел 2.8.2)
- Методы:
- Статический анализ кода (SAST) — SonarQube, Checkmarx
- Динамический анализ (DAST) — OWASP ZAP, Burp Suite
- Тестирование на проникновение (Penetration Testing) — внешний аудитор (опционально)
- Проверяемые уязвимости:
- SQL-инъекции
- XSS (reflected, stored, DOM-based)
- CSRF
- Небезопасная десериализация
- Неправильная настройка CORS
- Утечка информации (error messages, stack traces)
- Слабые алгоритмы шифрования
- Недостаточная валидация входных данных
- Критерий:0 критических уязвимостей, 0 высоких уязвимостей
#### 6.1.1.5. Юзабилити-тестирование (Usability Testing)
- Цель:проверка требований к удобству использования (НФТ-5.1 — НФТ-5.6)
- Метод:тестирование с реальными пользователями (5-10 человек)
- Сценарии:выполнение типичных задач (оформление заказа, использование конструктора, поиск товара)
- Метрики:
- Время выполнения задачи (task completion time)
- Количество ошибок (errors per task)
- Удовлетворённость (SUS — System Usability Scale, целевой балл ≥68)
- Критерий:≥80% пользователей успешно выполняют все задачи без помощи
#### 6.1.1.6. Регрессионное тестирование
- Цель:проверка, что новые изменения не сломали существующую функциональность
- Метод:автоматизированные тесты (unit, integration, E2E)
- Объём:все критические пути (оформление заказа, конструктор, управление заявками)
- Критерий:100% прохождение автоматизированных тестов
#### 6.1.1.7. Приёмочное тестирование (UAT — User Acceptance Testing)
- Цель:подтверждение, что система соответствует требованиям заказчика
- Исполнители:представители заказчика (менеджеры, мастера, бухгалтер)
- Сценарии:бизнес-сценарии из отчёта ИТК3
- Критерий:все критические сценарии пройдены, все серьёзные дефекты исправлены
### 6.2. Критерии приёмки
6.2.1.Для приёмки системы в эксплуатацию должны быть выполнены следующие условия:
#### 6.2.1.1. Функциональные критерии
- Все функциональные требования (ФТ-1.1 — ФТ-6.9) реализованы и протестированы
- Все пользовательские истории (ПТ-Г-1 — ПТ-Б-6) реализованы
- Все варианты использования (UC-01 — UC-06) протестированы
- Нет открытых критических (Critical) и высоких (High) дефектов
#### 6.2.1.2. Производительные критерии
- Время отклика ≤2 секунды для 95% запросов при 100 concurrent users
- Время отклика конструктора ≤1 секунды
- Время оформления заказа ≤5 секунд
- Пропускная способность ≥50 запросов/секунду
- Доступность ≥99.5% в течение 72 часов тестирования
#### 6.2.1.3. Критерии надёжности
- Система продолжает работать при недоступности Redis (graceful degradation)
- Восстановление из резервной копии ≤30 минут
- Резервное копирование работает ежедневно
- Журнал аудита фиксирует все критические операции
#### 6.2.1.4. Критерии безопасности
- Все пароли хэшированы (bcrypt/Argon2)
- Весь трафик по HTTPS (TLS 1.2+)
- Нет SQL-инъекций, XSS, CSRF уязвимостей
- Аутентификация обязательна для защищённых операций
- Авторизация на основе ролей работает корректно
- Ограничение попыток входа работает (5 попыток → блокировка 15 минут)
#### 6.2.1.5. Критерии документации
- Руководство пользователя готово и охватывает все роли
- Руководство администратора содержит инструкции по установке, настройке, мониторингу
- Руководство разработчика содержит архитектурные решения, описание кода
- Инсталляционное руководство позволяет развернуть систему с нуля
- API-документация (Swagger) актуальна и полна
- План тестирования и отчёт о тестировании предоставлены
#### 6.2.1.6. Критерии развёртывания
- Система развёрнута на production-сервере
- Docker-образы собраны и доступны в Registry
- Nginx настроен как reverse proxy с HTTPS
- SSL-сертификат установлен и действителен
- Мониторинг настроен (логи, метрики, алерты)
- Резервное копирование настроено и протестировано
- CI/CD пайплайн работает (автоматические сборки, тесты)
#### 6.2.1.7. Критерии обучения
- Проведено обучение администратора (2-4 часа)
- Проведено обучение менеджеров и мастеров (2-4 часа)
- Предоставлены видеоуроки или подробные инструкции
- Создана база знаний (FAQ, Knowledge Base)
6.2.2.Приёмка проводится комиссией в составе:
- Представитель заказчика (руководитель проекта)
- Архитектор/ведущий разработчик
- Project Manager
- QA-инженер
- DevOps-инженер
6.2.3.Результат приёмки оформляется протоколом с указанием:
- Списка выполненных требований
- Списка оставшихся дефектов (если есть) и плана их исправления
- Даты ввода в эксплуатацию
- Подписей всех членов комиссии
6.2.4.Система считается принятой, если:
- Все обязательные (Must-have) требования реализованы
- Нет открытых критических и высоких дефектов
- Все критерии приёмки (6.2.1) выполнены
- Заказчик подписал акт приёмки-передачи
