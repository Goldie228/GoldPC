# ADR-0002: Выбор PostgreSQL как основной СУБД

## Статус
Принято

## Контекст
Для проекта GoldPC необходимо выбрать СУБД для хранения данных о товарах, заказах, пользователях, услугах и гарантийных талонах.

### Требования
- ACID-транзакции для финансовых операций
- Сложные запросы (JOIN, агрегации) для отчётов
- JSON-поля для спецификаций товаров
- Полный текстовый поиск (поиск по каталогу)
- Надёжность и резервное копирование

### Ограничения
- Open-source решение
- Поддержка в .NET ecosystem
- Возможность масштабирования

### Текущее состояние проекта
Проект уже использует PostgreSQL:
- **Npgsql.EntityFrameworkCore.PostgreSQL 8.0.11** — провайдер EF Core
- **AspNetCore.HealthChecks.NpgSql 8.0.1** — health checks
- **Миграции** настроены в CatalogService и других сервисах
- **Connection strings** настроены в `appsettings.json`

```
<PackageReference Include="Npgsql.EntityFrameworkCore.PostgreSQL" Version="8.0.11" />
<PackageReference Include="AspNetCore.HealthChecks.NpgSql" Version="8.0.1" />
```

## Решение
Выбрана **PostgreSQL 16** как основная СУБД.

### Обоснование
1. **JSONB** — нативная поддержка JSON для хранения спецификаций товаров без отдельной таблицы
2. **Полнотекстовый поиск** — встроенный tsvector для поиска по каталогу
3. **ACID** — надёжность транзакций для заказов и платежей
4. **Расширения** — pg_cron для планирования, pg_stat_statements для мониторинга
5. **EF Core** — отличная поддержка через Npgsql провайдер
6. **Open Source** — бесплатная лицензия, активное сообщество

### Архитектура данных
- **Микросервисная структура** — каждый сервис имеет свою схему/базу данных
- **Shared DbContext** — общий контекст для типичных сущностей
- **Миграции** — управление схемой через EF Core Migrations

## Последствия

### Положительные
- ✅ Бесплатная лицензия (нет затрат на SQL Server)
- ✅ Высокая производительность для сложных запросов
- ✅ JSONB для гибких схем товаров (разные характеристики для разных категорий)
- ✅ Отличная поддержка в .NET через Npgsql
- ✅ Полнотекстовый поиск без внешних инструментов
- ✅ Активное сообщество и документация

### Отрицательные
- ⚠️ Нет встроенной репликации (настраивается отдельно)
- ⚠️ Требуется опыт администрирования для production
- ⚠️ Различия в SQL синтаксисе от SQL Server

### Риски
- **Средний риск** — при росте нагрузки потребуется оптимизация запросов
- Митигация: индексы, explain analyze, connection pooling

## Альтернативы

| Альтернатива | Плюсы | Минусы | Причина отклонения |
|--------------|-------|--------|---------------------|
| **SQL Server** | Интеграция с .NET, Tooling | Лицензионные затраты, привязка к Windows/Linux | Стоимость лицензий для production |
| **MySQL** | Популярность, простота | Слабее JSON поддержка, нет полнотекстового поиска | Меньше возможностей для проекта |
| **MongoDB** | Гибкая схема, JSON-native | Нет ACID по умолчанию, нет сложных JOIN | Не подходит для транзакционных данных |
